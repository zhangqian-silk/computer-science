# System Design

本目录用于系统设计知识的结构化沉淀，目标是形成可复用的方法、可迁移的能力模型与可落地的业务专题分析。

---

## 目标与边界

- 以文档为主，强调架构思考过程、边界定义与技术取舍依据。
- 代码仅作为教学级 demo，用于验证或解释关键机制，不以生产可用为目标。
- 关注「为什么这样设计」与「在什么约束下这样设计」，而非实现细节堆叠。

---

## 信息架构

采用五层结构组织内容，避免不同抽象层级混写。

| 层级 | 核心问题 | 内容类型 | 典型产出 |
| :--- | :--- | :--- | :--- |
| 方法论层 | 如何分析和拆解系统设计问题 | 通用分析框架、设计原则、权衡模型 | 方法论文档 |
| 通用机制层 | 分布式系统的基础机制如何设计 | 高可用、高并发、一致性、安全、幂等等 | 原理与模式文档 |
| 组件能力层 | 可跨业务复用的独立组件如何设计 | 规则引擎、限流器、短链、调度、缓存等 | 组件设计文档 + demo |
| 工程治理层 | 方案如何稳定上线与持续演进 | 可观测性、压测、灰度、回滚、SLA、故障演练 | 治理规范与实践手册 |
| 业务专题层 | 具体业务在真实约束下如何组合能力 | 营销、支付、广告、IM、电商等 | 业务架构专题文档 |

## 分类与落位规则

- 能脱离业务语境独立成立的内容，放在「方法论层」或「通用机制层」。
- 能被多个业务直接复用的模块化能力，放在「组件能力层」。
- 组件能力层文档需提供 demo 代码示例，用于解释关键机制与设计取舍。
- 涉及偏独立的算法、数据结构等基础内容，优先在 `fundamentals/` 中独立实现；`system_design/` 侧重实践与方案编排，并引用相关模块。
- 补充知识点时参考 `rule_engine` 模块组织方式：主要知识点集中在 `rule_engine.md`，重点内容或负责模块拆分到独立文件（如 `rete.md`）。
- 面向上线质量、运行稳定性、演进效率的共性实践，放在「工程治理层」。
- 强依赖业务对象、业务指标与业务流程的内容，放在「业务专题层」。
- 新文档应优先复用上层结论，减少同类概念在不同专题重复定义。

## 建议阅读顺序

- 先读方法论层，建立统一分析框架。
- 再读通用机制层，理解通用技术机制。
- 然后读组件能力层，掌握可复用设计单元。
- 再读工程治理层，理解从设计到上线的控制面。
- 最后读业务专题层，理解在具体约束下的组合落地。
