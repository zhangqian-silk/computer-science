# Transformer Extensions：从长上下文到视觉与多模态扩展

本文讨论 Transformer 在长上下文、高效率、视觉与多模态方向上的主要扩展路线，重点回答“标准 Transformer 主干如何被改造成更长上下文、更高效率、视觉模型与多模态系统”。如果希望先理解 block 结构、三类结构路线、训练目标与推理骨架，可先阅读 [transformer.md](./transformer.md)。

---

## 一、为什么需要扩展路线

标准 Transformer 非常强大，但它并不是没有代价。随着模型规模、上下文长度与应用模态持续扩展，研究重点很快从“Transformer 是否有效”转向了另一个问题：

**如何在尽量保留其统一主干的前提下，降低代价、扩大适用范围，并把它迁移到文本之外的模态。**

因此，Transformer 的后续演化大致沿 3 条主线推进：

- **效率改进**：让 attention、缓存与训练过程更便宜；
- **上下文扩展**：让模型更稳定地处理更长序列；
- **模态扩展**：把图像、音频等对象也改写成 token 序列纳入同一主干。

---

## 二、高效变体：让标准主干更便宜

标准 self-attention 的核心瓶颈，在于长度为 $n$ 的序列通常需要构造一个 $n\times n$ 的分数矩阵。于是，计算和显存代价都会随着序列长度明显上升。

围绕这一问题，后续高效变体大致可以分成以下几类：

| 方向 | 核心思路 | 主要目标 |
| --- | --- | --- |
| 高效 attention | 优化 attention 的访存组织与 kernel 实现 | 提升训练与推理效率 |
| 稀疏 attention | 只让部分位置彼此交互，而非全量两两相连 | 降低长序列成本 |
| 线性 attention | 通过核技巧或近似改写 attention 形式 | 缓解二次复杂度 |
| FFN / 激活改造 | 采用 SwiGLU、GeGLU 等结构 | 提升参数利用率 |
| 参数高效适配 | LoRA、Adapter、Prefix Tuning 等 | 降低微调成本 |

这些方法背后的共同动机很一致：标准 Transformer 很强，但也很贵。因此，很多后续研究并不是推翻主干，而是在保留 block 结构的同时，重写其中最昂贵或最不灵活的部分。

### 高效 attention

高效 attention 的重点并不总是“改变数学定义”，很多时候更像是“在保持等价结果的前提下优化计算路径”。FlashAttention 一类方法就属于这一思路，它通过更适合 GPU 的块式计算与访存组织，显著降低中间激活开销。

这一方向说明，Transformer 的性能瓶颈有时并不只来自理论复杂度，也来自硬件上“数据怎么被搬运和缓存”。

### 稀疏 attention

稀疏 attention 的思路是：既然不是所有位置都同等重要，那就不必总让所有位置彼此全量交互。于是，模型会只保留局部窗口、跨块连接、锚点连接或少数全局 token。

它的收益在于降低长序列成本，但代价是：

- 路由规则需要额外设计；
- 若稀疏模式选得不好，真实重要依赖可能被截断；
- 工程实现也往往比标准 attention 更复杂。

### 线性 attention

线性 attention 的核心是在数学形式上进一步近似标准 attention，希望把随长度增长的主要代价降下来。它通常更强调理论复杂度改进，但也可能在表达能力或训练稳定性上引入额外权衡。

因此，这条路线的关键不在于“是否更先进”，而在于：在什么任务、什么长度区间、什么硬件约束下，这种近似是否值得。

## 三、长上下文：从窗口变大到真正可用

长上下文并不只是“把窗口尽可能开大”。更准确地说，它包含 3 个不同层面的问题：

1. attention 代价如何增长；
2. 位置机制能否稳定外推到更长序列；
3. 推理阶段的缓存是否还能在有限显存下被有效管理。

### 长上下文推理的代价来源

在自回归推理中，模型不仅要处理更长的 attention 路径，还要维护不断膨胀的 KV cache。因此，当上下文持续增大时，瓶颈通常会同时来自：

- attention 计算本身；
- 历史 key / value 的存储；
- 更重的访存与带宽压力。

也正因为如此，很多长上下文系统会把缓存管理视为首要优化对象，而不是简单把“窗口越开越大”当成唯一目标。

### 常见工程思路

围绕长上下文，工程上常见的思路包括：

- **滑动窗口**：只保留最近的若干 token；
- **关键位置保留**：优先保留长期重要的 token；
- **分层或分块缓存管理**：按层、按位置范围或按硬件资源精细分配；
- **压缩与量化**：用更低精度或更紧凑的形式保存历史状态。

其中滑动窗口最直观，但也最容易牺牲早期关键信息。因为语言序列不像普通缓存数据那样只讲“最近最常用”，很早之前的系统提示、角色设定、长文主线与关键实体，依然可能决定后续生成。

### 与位置机制的关系

长上下文稳定性并不只取决于缓存管理，还高度依赖位置机制本身。绝对位置、相对位置、RoPE 与各种外推缩放策略，都会影响模型面对超出训练长度序列时的行为。

这一部分更适合放在机制层继续阅读，因此：

- 位置机制总览见 [positional-encoding.md](../mechanism/positional-encoding.md)
- RoPE 专题见 [rope.md](../mechanism/rope.md)
- 长上下文位置专题见 [long-context-position.md](../mechanism/long-context-position.md)

## 四、Vision Transformer：把图像改写成 token 序列

Vision Transformer（ViT）的关键思想，不是重新发明一套完全不同的注意力公式，而是先把图像改写成 Transformer 能处理的 patch token 序列。

设输入图像为
$$
I \in \mathbb{R}^{H \times W \times C}
$$

若把图像切成大小为 $P \times P$ 的不重叠 patch，则 patch 数量为
$$
N = \frac{HW}{P^2}
$$

对第 $i$ 个 patch，先展平成向量
$$
x_i \in \mathbb{R}^{P^2C}
$$

再通过线性投影映射到模型维度 $d$：
$$
z_i = x_iW_E + b_E
$$

随后再加入位置编码，把整段 patch 序列送入标准 Transformer Encoder。这样一来，self-attention 的语义就从“词与词建立关系”变成了“图像区域与图像区域建立关系”。

### ViT 与 CNN 的直观差异

如果只抓住最关键差异，可以概括为：

| 架构 | 基本视野 | 全局关系建立方式 | 典型归纳偏置 |
| --- | --- | --- | --- |
| CNN | 局部卷积窗口 | 通过多层堆叠逐步扩大感受野 | 强局部性、平移等变性 |
| ViT | patch 级 token | 通过 self-attention 直接全局交互 | 归纳偏置较弱，更依赖数据与预训练 |

因此，ViT 的意义不只是“Transformer 也能做视觉”，而是它验证了一个更一般的命题：只要某种输入可以被改写成序列，Transformer 主干就有机会继续工作。

## 五、多模态连接器：从视觉 token 到语言模型

现代多模态大模型常见设计的起点是：一个视觉编码器先输出图像 token 表示，再通过一个可学习投影层把视觉表示映射到语言模型的隐藏空间。

设视觉编码器输出为
$$
Z_{\mathrm{img}} \in \mathbb{R}^{N_v \times d_v}
$$

语言模型期望输入维度为 $d_{\mathrm{lm}}$，则可引入投影矩阵
$$
W_{\mathrm{proj}} \in \mathbb{R}^{d_v \times d_{\mathrm{lm}}}
$$

得到对齐后的视觉 token：
$$
Z_{\mathrm{img}\rightarrow\mathrm{lm}} = Z_{\mathrm{img}}W_{\mathrm{proj}}
$$

再把这些视觉 token 与文本 token 在序列维拼接：
$$
Z_{\mathrm{multi}} = [Z_{\mathrm{img}\rightarrow\mathrm{lm}}; Z_{\mathrm{text}}]
$$

于是，decoder-only 语言模型虽然底层仍在做“预测下一个 token”，但它的前缀中已经包含了视觉信息，因此能够继续完成图像描述、看图问答等任务。

### 为什么需要连接器

连接器存在的根本原因是：视觉编码器输出的特征分布，与语言模型输入 token 的分布并不天然一致。因此，多模态系统不能只把图像特征“硬塞”给语言模型，而需要一个中间层完成对齐、压缩与筛选。

从这个角度看，连接器更像“跨模态翻译层”，而不是独立的新主干。

## 六、从标准主干到大模型系统

Transformer 的真正扩展能力，不在于某一个具体变体，而在于它提供了一个足够统一、足够模块化、又足够适合并行计算的主干。因此，后续系统几乎都沿着同一骨架继续扩大：

- 更大的参数规模；
- 更长的上下文窗口；
- 更复杂的训练配方；
- 更重的系统级推理优化；
- 更强的多模态接入能力。

若把后续最有代表性的模型路线压缩成一张表，可以概括为：

| 路线 | 主干保留方式 | 核心扩展点 |
| --- | --- | --- |
| BERT 路线 | 保留编码器 | 强化表示学习与理解任务 |
| GPT 路线 | 保留解码器 | 强化自回归生成与规模化语言建模 |
| T5 / BART 路线 | 保留 encoder-decoder | 强化条件生成与统一文本到文本建模 |
| ViT 与多模态路线 | 保留 token 化与 attention 主干 | 扩展到文本之外的模态 |

从今天回看，Transformer 的真正影响不是“提出了一个更强的 NLP 模型”，而是提供了一个足够统一的建模接口，因此它才能从翻译一路扩展到大语言模型、视觉模型与多模态系统。

---

## 七、小结

Transformer 扩展路线真正回答的，不是“标准主干是什么”，而是“在主干已经成立之后，如何把它变得更长、更快、更多模态、更适合系统部署”。

如果把本文压缩成 4 个关键词，则分别是：

- **高效化**：解决 attention 与训练代价；
- **长上下文**：解决窗口、位置外推与缓存管理；
- **视觉化**：把图像切成 patch token 纳入统一主干；
- **多模态化**：用连接器把不同模态映射到语言模型可接收的隐藏空间。

这也说明，标准 Transformer 更像统一主干的起点，而高效化、长上下文、视觉化与多模态化则构成其后续主要演化方向。
