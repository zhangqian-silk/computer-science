# 模型理解维度参考

> 本文件**不是用于直接填空的写作模板**，而是一个「模型分析维度清单」。
>
> 它的主要作用是：当你学习一个模型时，帮助你直观判断应该从哪些方面去理解它、拆解它、比较它。
>
> 使用时不必机械照搬所有标题，也不要求每个模型都覆盖全部维度；应根据模型特点择其要者展开，避免不同维度之间重复讨论同一问题。

## 如何使用这份清单

- 若目标是**快速建立整体认识**，优先回答四个问题：它要解决什么问题、它如何建模、它的核心结构是什么、它相比前代改进了什么。
- 若目标是**写一篇模型介绍文档**，应以“定义清楚、机制讲清、例子说明白”为主，而不是平均铺开所有维度。
- 若目标是**横向比较多个模型**，应固定比较口径，例如：建模假设、信息流方式、上下文范围、复杂度、典型失效模式。
- 若目标是**深入学习单个模型**，则应进一步补充：损失函数、训练稳定性、推理流程、工程代价与历史位置。
- 任何一节都应尽量回答三个问题：**是什么、为什么、怎么做**。

## 理论基石与建模范式

内容核心：*说明“模型是什么”“模型假设了什么”“模型把任务看成什么问题”。*

可从以下角度理解：

- **任务背景与设计动机**
	- 先说明它所面对的数据类型与任务类型，例如文本序列、图像网格、时间序列、图结构数据。
	- 再说明它试图解决的核心痛点是什么。（例如：ResNet 试图缓解深层网络训练困难，Transformer 试图改善 RNN 难并行、长依赖难建模的问题。）
	- 若模型是在某一类旧方法基础上提出，应明确指出旧方法的瓶颈是什么。

- **模型范式**
	- 描述模型所属的方法论范式与架构类型，避免混用不同维度分类。
	- 可选口径：**统计学习**（如 N-gram/CRF）、**神经网络**（如 RNN/CNN/Transformer）、**参数化 vs 非参数化**（如 MLP vs KNN）。
	- 若一个模型同时具有多重属性，应区分“它属于什么大类”与“它采用了什么具体机制”。

- **问题建模**
	- 如何将现实任务转化为数学定义？
	- **数学目标：** 任务的数学本质（如：极大似然估计 $MLE$、后验概率 $MAP$、序列到序列映射等）。
	- **函数映射：** 模型试图拟合的函数形式 $f: \mathcal{X} \to \mathcal{Y}$ 的基本定义。
	- *示例：* 是将文本生成建模为**条件概率预测** $P(w_t|w_{<t})$？还是将图像分类建模为**高维空间的流形划分**？
	- 若是概率模型，应写清联合分布、条件分布或似然函数。
	- 若是神经网络模型，应写清输入、状态、输出之间的映射关系。

- **信号建模**
	- **数据抽象：** 原始信号被抽象为何种形态？（如：一维时间序列、二维空间网格、三维点云、非欧几里得图结构）。
	- **时空属性：** 是否考虑信号的时间因果性或空间平移不变性？
	- **局部与全局关系：** 模型更擅长捕捉局部结构，还是更擅长建模远距离依赖？
	- **离散与连续：** 模型处理的是离散符号、连续值，还是两者的混合表征？

- **归纳偏置**
	- 模型为了换取泛化能力而做出的先验假设。
	- *CV：* 空间局部性、平移不变性。
	- *NLP：* 位置相关性、因果性约束。
	- *时序任务：* 马尔可夫性、状态递归更新、平稳性假设。
	- 写这一节时，重点不是罗列术语，而是说明这些偏置如何帮助模型减少搜索空间、提升泛化能力。

- **信号处理哲学：**
	- 模型是如何处理数据冗余的？（如：通过卷积提取局部特征，还是通过 Attention 捕捉全局相关性）。
	- 也可进一步说明：模型是偏向“压缩历史信息”，还是偏向“显式访问历史信息”。
	- 这一节常用于解释模型为何在某些任务上自然有效。

## 架构设计与表征机制

内容核心：*说明模型由什么组成，信息在其中如何表示、变换与传播。*

可从以下角度理解：

- **输入/输出表征 (Input/Output Space)：**
	- 数据的量化形态（离散 Token、连续稠密向量、多尺度特征图）。
	- **CV：** 图像缩放、归一化、或 Patch Embedding + 位置编码。
	- **NLP：** Word Embedding + Positional Encoding。
	- 还可补充输入预处理方式，以及输出空间的含义：类别分布、连续值、隐变量、生成序列等。

- **核心算子与变换 (Core Operators)：**
	- **数学实现：** 详细列出最核心的转换公式（如：卷积算子、注意力权重矩阵、递归隐藏状态更新）。
	- 应优先写最能体现模型本质的公式，而不是把所有层都展开。
	- 每个公式之后最好说明：输入是什么，输出是什么，它解决了什么问题。

- **层级结构与拓扑 (Topology)：**
	- 模型由哪些模块组成？（Backbone、Neck、Head、Encoder-Decoder）。
	- 数据流是**线性串行**、**分支并行**还是**循环嵌套**？
	- 是否存在残差、跳连或门控设计？
	- 若模型有明显的模块分工，应写清每个模块负责什么，而不是只报模块名称。

- **上下文窗口与有效依赖长度 (Context Window / Effective Dependency Length)：**
	- 模型理论上能看到多长的上下文？
	- 模型实际上能稳定利用多长的依赖？
	- 若“理论可见范围”与“实际有效范围”差异很大，应明确指出。

- **状态与记忆机制：**
	- 模型是否维护显式状态，如隐藏状态、记忆单元、缓存、键值对记忆？
	- 历史信息是被压缩进单一状态向量，还是通过注意力、外部记忆等方式被显式访问？
	- 这一点往往决定模型的长程依赖能力与推理方式。

## 驱动引擎：损失函数与优化动力学

内容核心：*说明模型如何从数据中得到参数，以及训练为何可能有效或失效。*

可从以下角度理解：

- **损失函数体系 (Loss Functions)：**
	定义模型“学习”的数学本质与评价准则。

	- **目标函数形式：** 给出精确数学表达式（Cross-Entropy, MSE, Contrastive Loss, IoU Loss 等），以及推导方式。
	- **优化变量与极值目标：** 明确优化对象与目标方向（如最小化经验风险、极大似然估计 $MLE$、最小化 KL 散度等）。
	- **偏差惩罚机制：** 说明该 Loss 主要惩罚哪类偏差（如预测不确定性、空间坐标偏移、表征相似度失配）。
	- **任务对应关系：**
		- **分类任务：** 交叉熵 (Cross-Entropy)、Focal Loss。
		- **回归任务（如检测/分割）：** $L_1/L_2$ Loss、IoU Loss (Intersection over Union)。
		- **生成任务：** 感知损失 (Perceptual Loss)、对抗损失 (GAN Loss)。
	- 如果模型的“训练目标”与“最终评价指标”并不一致，也应单独说明。

- **学习算法 (Optimization Mechanism)：**
	定义参数是如何从数据中被计算或提取出来的。

	- **求解类型：**
		- 解析解/非梯度：闭式解 (Closed-form)、迭代计数 (Counting)、期望最大化 (EM)
		- 梯度优化：一阶法 (SGD, AdamW)、二阶法 (L-BFGS)
	- **参数更新逻辑**
		- 如何处理时序依赖？
		- 如何处理大规模并行？
	- 若模型存在特殊训练技巧，如教师强制、负采样、对比采样、截断 BPTT，应在此处说明。

- **收敛动力学与干预技巧**
	分析训练过程中的潜在风险以及人为的稳定性干预。

	- **稳定性策略：** 解决“能不能练成”的问题。
		- 策略：学习率预热 (Warmup)、梯度裁剪 (Clipping)。
		- 动力学：梯度消失/爆炸风险、局部最优陷阱分析。

	- **泛化性约束：** 解决“练得好不好”的问题。

		- 正则化手段：Dropout、权重衰减 (Weight Decay)、平滑技术 (Smoothing)。
		- 数据侧干预：数据增强 (Augmentation) 对收敛曲面的重塑作用。
	- 若某个模型“理论上简单，但训练上很难”，这一节通常是理解它的关键。

## 工程性能与推理规格

内容核心：*说明模型在真实使用环境中如何推理，以及它的计算代价与边界条件。*

可从以下角度理解：

- **推理决策算法 (Inference Logic)：**
	- 如何从模型输出得到最终结论？（如：Greedy Decoding, NMS 框过滤, Softmax 概率阈值）。
	- 写清具体输出函数、后处理规则，以及推理时是否依赖缓存、搜索或采样。
	- 若训练与推理采用不同策略，例如训练时教师强制、推理时自回归展开，应明确说明。

- **失效模式与鲁棒性 (Failure Modes)：**
	- **边界崩溃：** 在什么极端情况下会产生幻觉、误判或性能剧降？（如：语料稀疏导致 N-gram 出现零概率、Transformer 的二次方计算瓶颈）。
	- **输入敏感性：** 对噪声、长度变化、分布外样本、小样本、长尾类别是否敏感？
	- **错误类型：** 它更容易“看不见关键信息”，还是更容易“过度联想”？

- **复杂度规格 (Complexity)：**
	- **时间/空间代价：** 算力需求 (FLOPs) 与显存占用。随输入规模 $N$ 增长的复杂度 $O(\cdot)$。
	- **训练 vs 推理：** 两者的复杂度是否对称？瓶颈出现在计算、访存，还是序列展开？
	- **部署约束：** 是否适合在线场景、边缘设备、小显存环境或高吞吐离线场景？

## 时代坐标与演进 (Historical Context & Evolution)

内容核心：*说明它在模型演化链条中的位置，以及它为什么重要。*

可从以下角度理解：

- **代际突破：** 它修正了前代模型的哪个“硬伤”？（如：ViT 修正了 CNN 感受野不足的问题）。
- **后继影响：** 该模型的核心思想衍生出了哪些变体？当前是否已被更先进的架构（如 SSoE, State Space Models）取代？
	- 写这一节时，重点不在于罗列时间线，而在于说明“它引入了什么新假设或新机制”。
	- 若一个模型今天已不再是最强方案，也应说明它为何仍值得学习，例如它奠定了某类后续方法的基本思想。

## 使用建议

- 如果是**入门理解**，优先回答：它想解决什么问题、如何建模、核心结构是什么、相比前代进步在哪里。
- 如果是**深入分析**，再补充：损失函数、优化机制、复杂度、失效模式与工程约束。
- 如果是**横向比较**，可固定其中几个维度，例如：建模方式、归纳偏置、上下文范围、复杂度、历史位置。
- 如果是**写单篇模型笔记**，应围绕模型最有辨识度的部分组织内容，而不是平均分配篇幅。

## 推荐输出形态

- 对于多数模型，建议最终文档至少包含：**定义与假设、核心机制、训练与推理、最小例子、优势与局限、历史关系**。
- 若模型数学性较强，可先给**符号表**，再展开公式。
- 若模型机制较复杂，可优先使用**表格**解释模块、状态变化、模型对比，而不是单纯堆叠段落。
- 若模型最难理解的部分在“它到底做了什么”，应加入一个**最小推演例子**，用极小输入把中间状态变化写清楚。
- 写作时应避免两类问题：
	- **只有结论，没有机制。**
	- **只有公式，没有解释。**
