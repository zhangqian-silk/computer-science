# 经典编译流水线

经典编译流水线最适合用 C / C++ 作为主线理解。它的目标是把源代码提前翻译成某个操作系统和 CPU 可以直接执行的本地机器码，这类方式通常称为 AOT（Ahead-of-Time Compilation，预先编译）。

一条典型链路如下：

```text
hello.c
  ↓ 预处理
hello.i
  ↓ 编译
hello.s
  ↓ 汇编
hello.o
  ↓ 链接
hello / hello.exe
```

---

## 阶段总览

| 阶段 | 输入 | 输出 | 核心职责 |
|------|------|------|----------|
| 预处理 | 源文件 | 预处理后的源文件 | 处理 `#include`、`#define`、条件编译、注释 |
| 编译 | 预处理后的源文件 | 汇编代码或中间目标 | 分析语法语义，生成 IR，优化并输出低层表示 |
| 汇编 | 汇编代码 | 目标文件 | 把助记符翻译成机器码，并记录符号和重定位信息 |
| 链接 | 多个目标文件和库 | 可执行文件或共享库 | 解析符号、修正地址、合并代码和数据 |

这四个阶段并不只是「格式转换」。越靠前的阶段越关注语言规则，越靠后的阶段越关注机器、文件格式和地址。

---

## 预处理

预处理器是一个相对机械的文本处理阶段。它通常不理解完整的 C 语言语义，只识别预处理指令。

例如：

```c
#include <stdio.h>
#define PI 3.14

int main() {
    double area = PI * 2 * 2;
    printf("area = %f\n", area);
    return 0;
}
```

预处理阶段会做几件事：

- 处理 `#include`：把头文件内容展开到当前翻译单元中。
- 处理 `#define`：执行宏替换。
- 处理条件编译：根据 `#if`、`#ifdef`、`#ifndef` 等决定保留哪些文本。
- 删除注释：注释不会进入后续编译阶段。

经过简化后，可以理解为：

```c
int printf(const char *format, ...);

int main() {
    double area = 3.14 * 2 * 2;
    printf("area = %f\n", area);
    return 0;
}
```

### `#include` 是全量复制吗

从预处理器视角看，`#include` 更接近**全量文本包含**，而不是「只复制实际使用的函数」。原因是这一阶段还没有进入完整的语法和语义分析，预处理器不知道后续代码究竟会调用哪些函数。

这也解释了为什么头文件需要防重复包含：

```c
#ifndef MATH_UTIL_H
#define MATH_UTIL_H

int add(int a, int b);

#endif
```

如果没有 include guard 或 `#pragma once`，多个头文件层层包含同一个头文件时，预处理结果中可能出现重复声明或重复定义。

## 编译

编译阶段是整条流水线中最复杂的一段。它会把预处理后的源代码转成更适合机器执行的低层表示。

内部通常还会继续拆分为：

```text
字符流
  ↓
Token 序列
  ↓
AST
  ↓
类型检查后的语义树
  ↓
IR / SSA
  ↓
优化后的 IR
  ↓
汇编或目标代码
```

例如：

```c
double area = 3.14 * 2 * 2;
```

编译器可能会先把它识别为变量声明和初始化语句，然后在优化阶段做常量折叠，把 `3.14 * 2 * 2` 直接计算为 `12.56`。这样运行时 CPU 不需要再执行这次乘法。

常见错误也会在这个大阶段中被发现：

| 示例 | 更常见的发现阶段 | 原因 |
|------|------------------|------|
| `int a = ;` | 语法分析 | 缺少表达式，无法构建合法语法树 |
| `inr a = 1;` | 语法或语义分析 | `inr` 可能先被识别为标识符，但语义上不能作为类型使用 |
| `"apple" * 3` | 语义分析 | 字符串和整数不能按当前语言规则相乘 |
| `int b = a / 0;` | 不一定是编译错误 | 类型上可能合法，运行时才可能触发除零异常；优化器可能给警告 |

## 汇编

汇编器负责把汇编语言翻译成目标机器码。

汇编语言已经接近 CPU 指令，例如：

```asm
movl $16, -4(%rbp)
call printf
```

汇编器会把这些助记符翻译成二进制指令，并生成目标文件。目标文件中通常包含：

- `.text`：机器指令。
- `.data`：已初始化全局数据。
- `.bss`：未初始化或零初始化全局数据。
- 符号表：记录当前文件定义和引用了哪些符号。
- 重定位表：记录哪些地址还需要链接器修正。

目标文件已经包含 CPU 能执行的机器码，但它通常还不是完整程序。比如 `printf` 的真实实现可能不在当前目标文件中。

---

## 链接

链接器负责把多个目标文件和库组织成最终产物。

例如：

```text
main.o
math.o
libc.so / libc.a
  ↓
可执行文件
```

它主要完成三件事：

- 符号解析：找到 `printf`、`add` 等符号的真实定义。
- 地址重定位：把占位地址修正成最终地址或运行时可解析结构。
- 段合并与文件生成：把多个目标文件中的 `.text`、`.data` 等合并为可执行文件格式。

链接是从「单个翻译单元」走向「完整程序」的关键阶段。没有链接，多个 `.o` 文件之间只知道彼此的名字，不知道彼此真正位于哪里。

---

## 平台相关性

AOT 编译的结果通常和目标平台强绑定，至少包括：

- CPU 架构：x86-64、ARM64、RISC-V。
- 操作系统：Linux、Windows、macOS。
- 可执行文件格式：ELF、PE、Mach-O。
- ABI：函数调用约定、寄存器使用、栈布局、返回值传递方式。

因此，在 Windows 上生成的 `.exe` 不能直接拿到 macOS 上运行。它们不仅可执行文件格式不同，系统调用接口、动态库机制、加载器行为也不同。

## 常见命令观察

可以用 GCC 观察经典流水线：

```bash
gcc -E hello.c -o hello.i
gcc -S hello.i -o hello.s
gcc -c hello.s -o hello.o
gcc hello.o -o hello
```

也可以查看中间产物：

```bash
readelf -S hello.o
readelf -s hello.o
objdump -d hello.o
```

这几类工具分别帮助观察：

- 预处理后的文本长什么样。
- 汇编指令如何表达程序逻辑。
- 目标文件里有哪些 section 和符号。
- 哪些外部符号还等待链接器解决。
